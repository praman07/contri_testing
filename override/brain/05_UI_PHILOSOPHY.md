# UI Philosophy

The UI exists to make context, intent, approvals, evidence, and next actions visible. It must not become decoration.

## Every pixel earns its place

A UI element belongs only if it helps the user understand state, make a decision, approve an action, inspect evidence, or resume work. Remove anything that exists only to look advanced.

## Dynamic task-specific workspaces

Override should reshape the interface around the current task. The UI is not one permanent dashboard; it is a living workspace.

## Prohibited UI patterns

- Meaningless dashboards.
- Decorative futuristic panels.
- Fake telemetry.
- Percentages without meaning.
- Animated AI aesthetics that do not support decisions.
- Hiding risk or approval state.

## Information hierarchy

1. Active goal and current context.
2. Recommended next action.
3. Required approvals and risks.
4. Evidence and verification.
5. Relevant memory and history.
6. Secondary telemetry only when useful.

## Workspaces

Developer workspace: files, diffs, tests, logs, branch, services, plan, verification.

Learning workspace: goal, materials, weak areas, schedule, practice, progress.

Research workspace: sources, claims, citations, contradictions, notes, diagrams.

Career workspace: roles, fit, resume variants, cover letters, submissions, follow-ups.

Meeting workspace: agenda, participants, notes, decisions, tasks, project links.

Diagram workspace: source context, generated diagram, relationships, revisions, export.
