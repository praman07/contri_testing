# Product Philosophy

Override exists because users should not have to translate every intention into mechanical software steps. Computers should understand the work context, coordinate tools, and reduce friction while keeping humans in control.

## Problems Override solves

- AI systems forget the workspace.
- Chatbots wait for prompts instead of understanding ongoing work.
- Automation tools execute steps without understanding why.
- Developer tools operate inside one app and miss the broader workflow.
- Users lose context across windows, projects, meetings, and sessions.

## Human-first philosophy

Override assists; it does not replace judgment. The user owns the goal, the data, the approvals, and the final decision. Override may recommend, plan, execute approved actions, verify, and remember. It must not manipulate, pressure, or hide uncertainty.

## Local-first philosophy

Core context, memory, and workspace understanding should remain local whenever practical. Cloud models and external APIs are optional providers, not architectural dependencies.

## Transparency and trust

Trust comes from clear intent, explainable reasoning, visible approvals, auditable history, and reliable verification. Override must be able to answer: what do you know, why do you recommend this, what will you do, what did you do, and how do you know it worked?

## Non-goals

Override must never become a chatbot wrapper, vendor-specific agent, telemetry dashboard, decorative assistant, OS replacement, uncontrolled automation tool, or closed memory silo.

## Product principles

1. Context before command.
2. Human control before autonomy.
3. Verification before confidence.
4. Local-first before cloud dependence.
5. Useful silence before performative chatter.
6. Task-specific UI before generic dashboards.
7. Durable memory before chat history.
8. Provider abstraction before tool coupling.
9. Honest uncertainty before fabricated certainty.
